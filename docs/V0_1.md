# AndSpace v0.1 — Foundation Layer

v0.1 builds on the **approved** Phase 0 Tauri + xterm.js terminal. The current
focus is foundation infrastructure for Command Guard and ⌘E handoff later, not
those features themselves.

## In scope (milestone 1)

- zsh OSC integration (`src-tauri/shell-integration/andspace.zsh`)
- Per-pane metadata: cwd, last command, exit code, output boundary, timestamps
- Simple per-pane command history in the frontend store
- Diagnostic logging of lifecycle events to `/tmp/andspace-diag.log`
- Docs: this file and `docs/SHELL_INTEGRATION.md`

## Milestone 2 — auto-source zsh integration (done)

- `ZDOTDIR` bootstrap under `shell-integration/zdotdir/` (does not modify `~/.zshrc`)
- `shell-autoload` diagnostic on PTY create

## Milestone 3 — ANDSPACE rules parser + resolver (done)

- Rust parser/resolver in `src-tauri/src/rules.rs`
- Tauri command: `load_rules_for_cwd(cwd: String)`
- Project rules: `./ANDSPACE.md`
- User-global rules: `~/.andspace/rules.md`
- Built-in protected/dangerous defaults
- Precedence: project > user-global > built-in
- Frontend helper/store path requests resolved rules for the active pane cwd
- Diagnostic rule counts are written to `/tmp/andspace-diag.log`
- Full format docs: `docs/ANDSPACE_RULES.md`

## Milestone 4 — Command Guard dry-run evaluator (done)

- Rust evaluator in `src-tauri/src/command_guard.rs`
- Tauri command: `evaluate_command_guard`
- Matching order: allowed > dangerous > protected
- Dry-run diagnostics only; terminal behavior is not blocked or interrupted
- Frontend stores a short in-memory per-pane guard evaluation history
- Diagnostic results are written to `/tmp/andspace-diag.log`
- Full behavior docs: `docs/COMMAND_GUARD.md`

## Milestone 5 — zsh Command Guard pre-execution gate (done)

- zsh `accept-line` wrapper in `src-tauri/shell-integration/andspace.zsh`
- Checks `$BUFFER` before zsh executes the command
- Safe and allowed commands run normally
- Protected commands ask inline `Run once? [y/N]`
- Dangerous commands require exact `run`
- Temporary shell-side matcher mirrors allowed > dangerous > protected
- Pre-execution diagnostics are written to `/tmp/andspace-diag.log`

## Milestone 6 — Command Guard hardening + rule sync (done)

- Rust resolver/evaluator documented as canonical for app-side logic and tests
- zsh matcher documented as intentional temporary v0.1 blocking path
- zsh diagnostics include `matcher_impl=zsh`
- zsh matcher hardening: empty/comment-only rules ignored, invalid regex rules do not break input
- Rust parity coverage for the same command fixture used by zsh verification
- Dev verification script: `scripts/verify-command-guard-zsh.sh`

## Milestone 7 — Command Guard confirmation UI (done)

- zsh pre-execution gate emits OSC `guard-request` for protected/dangerous commands
- Native AndSpace overlay shows command, matched rule, and source
- Protected commands support `Cancel` / `Run once`
- Dangerous commands require exact `run` before `Run command`
- Rust response bridge writes `/tmp/andspace-guard-<request>.response`
- UI request/action diagnostics are written to `/tmp/andspace-diag.log`
- Missing or timed-out UI responses default to cancel

## Explicitly out of scope

- Sidebar, command palette, AI UI
- ⌘E handoff UI
- Project / file features
- Background watchers
- Writing to the user's `~/.zshrc` on disk

## Next milestones (not started)

- Rust-backed synchronous Command Guard bridge for the pre-execution gate
- Additional Command Guard product polish beyond the compact confirmation overlay
- ⌘E handoff payload from last command + cwd
- Binary IPC channel for PTY output if throughput regresses

## Phase 0 carry-over

- **#11 background-tab throttling** remains a documented WKWebView limitation.
- Position AndSpace as a **workflow terminal**, not a raw Ghostty competitor.
